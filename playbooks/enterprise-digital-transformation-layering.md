# 传统企业数字化转型分层判断 Playbook

## 适用场景

用于判断传统制造、渠道、销售、CRM、ERP、供应链等场景中的数字化需求应该放在哪一层，以及如何避免“买系统 / 堆功能 / ERP 无限二开”带来的长期失控。

本 Playbook 来自多条打捞资料的综合沉淀：企业 Agent 语义层、FDE 引入节奏、王自如 / 格力数字化改革口播整理等。它不是某一家企业的复盘结论，而是可复用的诊断框架。

## 核心判断

传统企业数字化不是软件采购问题，而是四件事的组合：

1. 一把手授权：是否有人能推动跨部门流程与权责调整。
2. 系统分层：是否区分前台流程、语义中间层和主数据沉淀层。
3. 数据治理：是否统一业务概念、指标口径、主数据和异常处理。
4. 阶段推进：是否先线上化，再数据化，最后自动化 / AI 化。

## 三层架构

### 1. 业务应用层 / 前台操作层

承载高频变化、人员日常操作和业务试错。

常见对象：

- CRM 前台
- 渠道系统
- 经销商后台
- 订单中台
- 活动 / 促销系统
- 外勤 App
- 飞书审批 / 群自动化入口

判断标准：

- 规则经常变。
- 用户交互频繁。
- 需要快速上线和试错。
- 失败成本相对可控。

### 2. 语义中间层 / 规则转换层

承载业务概念、口径、字段映射、规则校验、权限护栏、异常拦截。

常见对象：

- 指标口径字典
- 客户 / 线索 / 终端客户保护等业务对象定义
- 跨系统字段映射
- 数据同步校验规则
- AI Agent 中间态
- 权限与动作白名单
- 异常单据拦截逻辑

判断标准：

- 需要跨部门共识。
- 需要把业务语言翻译成系统动作。
- 需要审计、追溯、校验和回滚。
- AI / 自动化不能直接端到端执行，需要中间态。

### 3. 主数据 / ERP / 权威沉淀层

承载最终闭环后的权威数据和稳定核算。

常见对象：

- 客户主数据
- 供应商主数据
- 商品 / SKU
- 组织架构
- 合同与财务结果
- 库存结果
- 成本、收入、结算、核算数据

判断标准：

- 要强一致。
- 要可审计。
- 变化频率低。
- 出错影响财务、供应链或经营决策。
- 不适合频繁承载业务玩法。

## 需求分层检查清单

收到一个数字化 / AI / 自动化需求时，先问：

1. 这个需求是在改变用户操作，还是改变权威数据？
2. 其中哪些是稳定主数据，哪些是流程过程数据，哪些是临时规则？
3. 是否存在多个部门对同一指标 / 对象理解不同？
4. 是否需要权限判断、异常拦截、人审确认或日志审计？
5. 如果业务规则下个月变化，是否会牵动 ERP / 财务 / 库存核心链路？
6. 当前痛点是“没有系统”，还是“口径不统一 / 权责不清 / 数据不可信”？
7. AI 介入时，是否有中间态可检查，而不是直接生成最终动作？

## 典型反模式

### 反模式 1：把 ERP 当万能业务系统

表现：所有流程、促销、返利、渠道规则、临时审批都往 ERP 里塞。

后果：系统越来越重，升级困难，月结 / 对账 / 核算变慢，业务变化拖不动。

修正：ERP 只沉淀最终闭环后的权威数据，高频变化放到业务层，中间通过语义 / 数据校验层转换。

### 反模式 2：只做前台工具，不做语义层

表现：上了很多 App、小程序、自动化机器人，但客户、区域、产品、审批、指标口径没有统一。

后果：前台看似热闹，后台数据继续打架，AI 查数和自动化执行都不稳定。

修正：补业务对象定义、字段映射、指标口径、权限规则、异常处理和回归测试。

### 反模式 3：用短期营收考核基础设施

表现：主数据治理、语义层、权限中心、数据同步等项目三个月没有直接增收就被砍。

后果：永远停留在碎片化工具，组织级复用能力起不来。

修正：基础设施看里程碑、可复用资产、口径统一、异常减少、扩展成本下降；应用项目再看活跃、效率和业务指标。

### 反模式 4：AI 直接穿透业务系统

表现：用户一句自然语言，AI 直接查库、写系统、发通知、改状态。

后果：权限、口径、误判、责任归属和回滚都不可控。

修正：用户原话 → 意图解析 → 业务对象识别 → 参数确认 → 权限检查 → 动作计划 → 人审 / 自动执行 → 日志审计。

## 推荐推进阶段

### 阶段 0：Excel 标准化 + AI 个人提效破冰

目标：先用低阻力方式收拢业务语言和个人使用习惯，不急着上重型系统。

优先做：

- 统一客户台账、报价、库存、订单、周报等高频 Excel 模板。
- 明确字段定义、统计口径、计算逻辑、必填项、责任人和校验规则。
- 鼓励员工在文档整理、表格处理、信息提炼、方案草拟、周报生成中使用 AI。
- 收集高频 AI 场景、好用 Prompt / Skill、失败案例和流程卡点。

这一阶段的意义不是把 Excel 当终局，而是把散落在个人表格和个人经验里的“明知识”先沉淀出来，为后续线上表单、低代码应用、数据中间层和内部 AI 平台打底。

### 阶段 1：行为线上化

目标：凡事留痕、线上可查。

优先做：申请、报销、报价、客户档案、订单、拜访、问题反馈等高频动作线上化。

不要一开始就追求复杂中台和全面重构。

阶段 0 中沉淀出的 Excel 模板，可以逐步迁移为线上表单、低代码应用或业务系统模块。

### 阶段 2：业务数据化

目标：统一对象、口径、主数据和跨系统链路。

优先做：客户、产品、区域、组织、指标、审批状态、异常类型等基础语义资产。

这阶段需要一把手授权，因为会触碰部门台账、信息特权和历史习惯。

### 阶段 3：数据业务化 / AI 自动化

目标：让数据反向驱动业务动作。

可做：预测、预警、自动分配任务、风险拦截、补货建议、信用冻结提醒、客户经营建议、AI Agent 工作流。

前提：前两阶段已经形成可信数据、稳定口径和可审计动作链。

## Excel / AI 个人提效阶段的治理要点

### Excel 标准化不是简单发模板

至少要同步定义：

- 字段含义
- 填写口径
- 数据来源
- 责任人
- 更新频率
- 校验规则
- 版本号和唯一入口

否则统一模板很快会重新分裂成多个个人版本。

### AI 个人提效不是自由试用到结束

需要同步沉淀：

- 高频使用场景
- 重度使用者
- 可复用 Prompt / Skill
- 常见失败案例
- 数据与权限卡点
- 值得平台化的动作

否则个人提效不会自然变成组织提效。

## 对 CRM AI 化的落地启发

CRM AI 化优先不要追求“AI 什么都能干”，而要先围绕高频对象建立语义层：

- 客户
- 线索
- 终端客户
- 保护申请
- 区域
- 产品线
- 销售负责人
- 审批状态
- 报价
- 拜访
- 风险提示

每个对象至少沉淀：业务定义、同义说法、数据来源、权限要求、常见歧义、是否允许写入、是否需要人审。

## 公开边界

可公开：

- 抽象方法
- 分层框架
- 检查清单
- 通用反模式
- 脱敏后的案例结构

不可公开：

- 内部系统截图
- 客户 / 供应商 / 人员信息
- 真实审批链路和权限细节
- 具体接口、字段、日志
- 未公开业务策略

## 概念对照与来源分级

推进数字化 / AI 转型时，可以用“原始概念 → 口语化包装 → 落地动作 → 验证状态”来管理知识来源，避免把短视频表达、行业共识和未验证传闻混在一起。

| 原始概念 | 口语化包装 | 落地动作 | 验证状态 |
| --- | --- | --- | --- |
| SAP Clean Core | ERP 只做数据沉淀 / 数据鱼塘 | 业务流程与临时规则移出 ERP，通过扩展或中间层同步闭环数据 | 公开资料可支持 |
| 中台 / 前中后台分层 | 业务应用层 → 语义中间层 → ERP 沉淀层 | 前台敏捷迭代，中间层做能力复用与规则转换，后台保持稳定 | 公开资料可支持 |
| BPR 业务流程再造 | 数字化是一把手工程 | 流程、权责、绩效和数据审计需要组织授权 | 理论方向可支持，具体组织设计需结合企业 |
| Ontology / 业务本体 | 业务概念、明知识 + 暗知识 | 建模客户、产品、订单、审批、区域、指标等对象和规则 | 公开资料可支持 |
| Semantic Layer | 语义中间层 / 可检查中间态 | 统一指标口径、字段映射、权限、安全、异常拦截 | 公开资料可支持 |
| AI 个人提效 | 员工 AI 孵化期 | 收集场景、Prompt、Skill、失败案例和重度用户 | 需要企业内部样本验证 |
| FDE / AI BP | 组织级 AI 工程化角色 | 负责语义层、Skill、工作流和业务闭环落地 | 概念可支持，具体 SOP 需追源 |
| Agent 自动化 | 数据业务化 / 自动调度 | 预测、预警、风控、任务分配、自动执行 | 需要权限、数据和业务闭环验证 |
| 国产模型优先 | 企业模型治理白名单 | 基于合规、成本、能力、可用性做模型评测 | 不能只依赖传闻，必须实测 |
