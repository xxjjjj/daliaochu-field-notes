# Skill to Loop Engineering Playbook

## 1. 适用场景

当一个 Agent Skill 已经能稳定完成某类任务，但仍依赖人工反复纠正、补充、提醒时，可以考虑从静态 Skill 升级为 Loop。

适合优先试点的任务：

- 高频、边界清晰；
- 有明确输入对象，例如资料、Issue、报错、审批消息、CRM 操作意图；
- 有可观察反馈，例如人工修改、评论纠正、标签变化、测试结果、业务确认；
- 错误成本可控，能回滚、能人审；
- 不涉及未经脱敏的客户数据或敏感流程公开沉淀。

不适合直接上 loop 的任务：

- 高风险审批、正式承诺、财务/法务/人事决策；
- 反馈信号模糊且不可验证；
- 输入含大量敏感数据但没有脱敏与权限边界；
- 失败后没有回滚机制；
- 只是为了“全自动”而没有明确业务收益。

## 2. 基本结构

一个可控的 Skill2Loop 至少包含六层：

1. **触发层**：定时任务、Webhook、飞书消息、GitHub Issue、CI 失败、CRM 事件等。
2. **执行层**：调用明确的 Skill / Playbook 处理单个工作对象。
3. **记录层**：把输入摘要、判断、动作、结果、版本号写入可追踪位置。
4. **反馈层**：收集人类修改、评论、reaction、label 变化、测试结果、业务确认。
5. **学习层**：把反馈归纳成可泛化规则，过滤一次性个例和噪声。
6. **变更层**：通过 PR、待审文档或人工确认更新原 Skill，而不是静默覆盖。

## 3. Inner Loop：先把事做稳

内循环只负责用当前 Skill 做事，不负责改规则。

推荐动作：

- 读取输入对象；
- 补齐必要上下文；
- 按 Skill 输出结构化结果；
- 标记版本号、来源、public_level、value_stage；
- 写入日志、note、experiment 或业务系统；
- 在输出中留下可被反馈层识别的 marker。

示例：

- 打捞处：收到资料 → 追源 → 摘要 → 判断公开等级 → 写 note → commit。
- Issue 分诊：新 issue → 查相似 issue → 打唯一分类 label → 评论说明 → 请求反馈。
- CRM AI：自然语言请求 → 识别意图 → 检查权限/字段 → 缺失则追问 → 可执行再写入。

## 4. Outer Loop：再让经验回写

外循环只负责学习和提出改进建议，不直接扩大权限或绕过人审。

推荐动作：

1. 定期读取最近一段时间的内循环结果。
2. 收集反馈信号：
   - 强信号：人工直接改结果、明确纠错、测试失败、业务拒绝；
   - 中信号：补充说明、追问、二次修改；
   - 弱信号：沉默、简单点赞、没有后续。
3. 判断是否存在可泛化经验。
4. 只把“类别级经验”写入规则，不记录孤立个例。
5. 生成 Skill / Playbook 更新草案。
6. 通过 PR 或待审文档给人确认。

## 5. 可泛化规则判断标准

可以写入 Skill 的经验：

- 同类问题重复出现；
- 有明确人工纠错证据；
- 能转成下一次可执行的判断条件；
- 不依赖某个具体客户、某条群聊、某个内部截图；
- 与现有规则不冲突，或能说明如何修订旧规则。

不应写入 Skill 的经验：

- “某一次答错了”但没有类别规律；
- 只适合一个人、一个客户、一个临时项目；
- 包含敏感原文、客户数据、内部系统细节；
- 只是口味偏好，没有稳定判断价值；
- 会导致 Agent 扩权、替人承诺或绕过审批。

## 6. 安全边界

- 关键 Skill 更新必须可审查、可回滚。
- 外循环默认只生成建议，不直接改生产规则。
- 涉及公开仓库时，只提交 public 或 sanitized 内容。
- 涉及内部数据时，优先写 private-only 或 needs-review，不推公开仓库。
- 对自动化执行设置预算、频率、最大迭代次数和失败停止条件。
- Maker / Checker 尽量分离：执行者不直接给自己做最终质量判定。

## 7. 最小可行实验

以打捞处为例：

1. 内循环：每条资料写一张 salvage card，并提交 GitHub。
2. 反馈记录：观察后续人工补充、修正、追问、文件修改。
3. 外循环：每周读取最近 notes/experiments/playbooks 的变更。
4. 归纳：提炼对模板、标签、公开边界、追源流程的改进建议。
5. 审查：生成 PR 或一份待确认的 playbook 更新。
6. 合并：确认后再进入主规则。

## 8. 当前建议

先不要把 Skill2Loop 做成大而全平台。优先选一个低风险场景验证：

- 打捞处学习沉淀 loop；
- 飞书自动化回复质量改进 loop；
- RPA/NC 报错诊断 playbook 改进 loop；
- CRM AI 意图识别与追问规则改进 loop。

验证指标可以很简单：返工是否减少、盲测是否减少、人工纠错是否被回写、下一次同类任务是否更稳。
