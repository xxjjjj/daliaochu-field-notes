# 离职/停合作员工资源回收清单 Playbook

## 1. 它是什么

当员工离职 / 外部合作终止 / 顾问合同时，**在第一时间识别并回收该员工在企业内部已经拿到的所有资源（账号、API Key、Token、订阅、权限、应用、bot）** 的标准化流程。

本 Playbook 是从打捞处群聊 2026-05-28 → 2026-05-29 一次真实的"准备停止与 6 位外部老师合作"事件中沉淀的工作流。

## 2. 解决什么问题

- 内部账号、API Key、订阅权限"谁拿过、谁还在用、有没有轮换"基本没人盯，员工离职时容易遗漏。
- 外部顾问/合作老师用的资源比正式员工更多（因为跨多个系统），但 Offboarding 流程通常只覆盖正式员工。
- 群聊里过去给过的 key/token 没记录，复查成本极高。
- 没有"按优先级回收"的标准化清单，导致操作时手忙脚乱。

## 3. 标准流程

### 3.1 第一步：圈定人员 + 时间范围

明确：

- **人员清单**：本次需要回收资源的员工 / 外部老师姓名清单
- **时间范围**：从该员工入职 / 第一次合作开始，到当下为止
- **建议时间起点**：至少往前 12 个月；如果 12 个月内有重大权限变更，按变更时间点倒推

### 3.2 第二步：搜索关键词扫描

对内部 IM（飞书 / 钉钉 / 企微 / Slack）的全部群聊做关键词扫描，覆盖：

```
账号 / 系统 / 应用 / 权限 / key / token / API / Secret /
Access Key / 密码 / 登录 / 接口 / 地址 / 配置 / bot / 机器人 /
飞书应用 / HubSpot / WhatsApp / Claude / 火山 / 数仓 / 阿里云 /
腾讯云 / AWS / GCP / Notion / Figma / GitHub
```

每个搜索词检索出该员工相关的消息条数，然后人工筛掉非资源类消息。

### 3.3 第三步：分类输出回收清单

按"可执行"维度，把所有线索整理成 5 类：

| 类别 | 回收动作 | 优先级判断依据 |
|------|----------|----------------|
| **账号** | 改密 / 禁用 / 注销 | 是否有共享账号风险、是否还有在用 |
| **API Key / Token** | 重新生成 key，旧 key 立即作废 | 是否暴露过、是否用在生产链路 |
| **订阅 / 资源包** | 取消订阅 / 转移所有权 / 退订 | 是否仍在自动续费 |
| **权限 / 角色** | 撤销角色 / 调整可见范围 | 是否有数据导出权限、是否有 admin 角色 |
| **应用 / bot / 工作流** | 转移所有权 / 删除 / 归档 | 是否在跑生产流程 |

每条记录包含：

- 资源名称
- 类型（账号 / Key / 订阅 / 权限 / 应用）
- 给出人 / 接收人
- 给的时间点
- 当前状态（在用 / 已停 / 不确定）
- 优先级（P0 立即 / P1 核实后处理 / P2 补充核实）
- 回收动作建议

### 3.4 第四步：按优先级执行

| 优先级 | 时间窗口 | 范围 |
|--------|----------|------|
| **P0 立即** | 当天 24 小时内 | 账号、密码、共享 key、生产链路 token |
| **P1 核实后处理** | 3 天内 | 订阅、付费资源包、跨系统权限 |
| **P2 补充核实** | 1 周内 | 一次性资源、试用账号、非生产链路 |

### 3.5 第五步：写入多维表格存档

把清单写入飞书多维表格，字段包括：

```
人员 | 资源类型 | 资源名称 | 给出时间 | 当前状态 |
优先级 | 回收动作 | 执行人 | 执行时间 | 复核人 | 备注
```

方便后续审计、复用、对比下一次回收。

## 4. 真实案例的关键发现

### 4.1 第一次扫描只能给出"线索"，不能直接用作"回收清单"

只做关键词扫描的输出是"信息线索汇总"，不是"停合作后的账号/密钥/资源回收清单"。两者区别：

| 线索汇总 | 可执行清单 |
|----------|------------|
| "Janice 提过火山子账号" | "火山子账号 Janice：立即改密、查 AK/SK、禁用或回收子账号" |
| "Shire 提过 Claude 试用账号" | "Claude 试用账号：登录审计，确认未续费，登录密码修改" |

**必须做第二遍加工**，把每条线索转成可执行的回收动作 + 优先级，否则操作人员无法落地。

### 4.2 时间范围要足够

只扫描"最近 3 个月"会漏掉大量历史给过的资源。建议起点至少 12 个月前，重要岗位（财务 / IT / 高管）的资源给过的历史可能超过 2 年。

### 4.3 关键词扫描会漏掉的资源类型

- **登录过的第三方 SaaS**：如果员工没在群聊里说"我登了 XXX"，扫描不到。建议同时看 SSO / IdP 的登录日志
- **个人手机号绑定的二次验证**：扫描不到，需要单独看账户设置
- **GitHub / GitLab Personal Access Token**：要单独审计仓库的 collaborator 列表
- **浏览器保存的密码**：要靠企业密码管理器（如 1Password / LastPass Teams）审计

## 5. 验证清单

- [ ] 所有"线索"已转成"可执行动作"
- [ ] P0 项已全部执行 + 复核
- [ ] P1 / P2 项已排期跟进
- [ ] 多维表格已存档 + 设置了责任人
- [ ] 给外部合作老师的资源已通知对方正式终止
- [ ] 涉及生产链路的 key 已轮换 + 验证旧 key 失效

## 6. 风险与边界

- **误回收风险**：如果对某条线索判断不准就回收，可能影响仍在运行的业务。建议 P0 / P1 都设"复核人"二次确认
- **沟通风险**：对外终止合作 + 回收资源是两个独立动作，建议先沟通再回收，避免法律纠纷
- **数据丢失风险**：如果员工个人账号下存了企业资料（云盘 / 邮件），回收前要先备份
- **凭证轮换风险**：核心链路 key 轮换要选业务低峰期，且准备好回滚方案
- **审计要求**：金融 / 医疗 / 政企客户对"谁在什么时间点停用了谁的资源"有合规审计要求，必须留痕

## 7. 复用条件

- 任何员工离职 / 顾问结束 / 外部合作终止场景都适用
- 建议每个季度做一次"全员资源活跃度审计"，提前发现沉睡资源

## 8. 已知替代方案

| 方案 | 适用场景 | 备注 |
|------|----------|------|
| 人工群聊搜索 + 多维表格 | 50 人以下中小企业 | 本 Playbook 主体方案 |
| 企业 IdP 自动回收（Okta / Azure AD） | 中大型企业 | 自动触发离职流程，无需群聊扫描 |
| SaaS 管理平台（如 Zluri / Productiv） | 100+ SaaS 订阅的中大型企业 | 自动识别资源使用情况，但成本高 |
| 第三方安全审计（如 KPMG / 德勤） | 金融 / 上市公司年度审计 | 外部合规要求时采用 |
