# 飞书话题群历史内容补入库 Playbook

## 目标

把打捞处话题群中已经产生、但尚未进入 `daliaochu-field-notes` 仓库的历史内容，按“资料型 / 实验型 / 方法型 / 风险边界”重新整理入库。

本 Playbook 适用于追溯近 2-3 个月历史消息，不做群聊原文搬运，只沉淀结构化摘要、价值判断、验证状态和下一步动作。

## 适用范围

优先处理以下内容：

1. 外部资料链接：文章、视频、GitHub、产品官网、论文、公开案例。
2. 技术 / 开源线索：需要追到 GitHub、官网、文档或 Demo 后再判断。
3. MVP / 实验记录：测试目标、输入材料、过程、结论、失败点、下一步。
4. 已多次验证的方法：沉淀到 `playbooks/`。
5. 公开边界、版权、数据安全、公司内部实践风险：沉淀到 `risks/`。

## 不直接公开的内容

以下内容不得原样进入公开仓库：

- 群聊原文、个人表达、未脱敏对话；
- 客户、供应商、真实业务数据；
- 内部系统截图、日志、字段、接口、流程细节；
- 未授权课件、付费内容、长段原文或逐字稿；
- 付费课程「八大心智」相关原文、课件结构、老师原话、课堂纪要或可替代课程交付的系统化笔记；
- 尚未确认公开边界的内部实践。

这些内容只能作为线索，转成脱敏后的方法、风险、实验设计或待审核条目。

## 默认处理流程

```text
拉取/导出历史消息
  ↓
按月份和话题初步分组
  ↓
识别内容类型
  ↓
去重：检查 notes / experiments / playbooks / risks 是否已存在
  ↓
追源：链接类、技术类资料尽量追到本体
  ↓
判断 public_level 与 value_stage
  ↓
写入对应目录
  ↓
git commit + push
  ↓
输出本轮补录清单与待追源清单
```

## 入库路径规则

- 资料型内容：`notes/YYYY-MM/YYYY-MM-DD-slug.md`
- MVP / 实验型内容：`experiments/YYYY-MM/YYYY-MM-DD-slug.md`
- 可复用方法：`playbooks/topic-name.md`
- 风险边界：`risks/topic-name.md`

## public_level 判断

- `public`：纯公开资料、自写摘要、无内部信息。
- `sanitized`：来源于群聊或内部讨论，但已去除人名、公司细节、客户、截图、字段、数据。
- `private-only`：包含真实内部系统、客户、业务数据、未脱敏流程，不进入公开仓库。
- `needs-review`：边界不确定，需要人工确认后再公开。

公开仓库只提交 `public` 和已脱敏的 `sanitized` 内容。

## value_stage 判断

- `待追源`：只有二手线索，尚未看到官网、仓库、文档、Demo 或原始资料。
- `学习理解`：已读基本资料，但尚未形成验证计划。
- `待验证`：价值方向明确，但缺少实测或业务样本。
- `可小实验`：可用低成本样本跑一次验证。
- `可业务试点`：已有初步验证，适合进入小范围业务场景。
- `已沉淀`：可复用为模板、Checklist、Playbook。
- `暂不建议`：风险、成本、边界或真实性问题较大。

## 近 2-3 个月补录策略

### 第一轮：建立历史消息清单

按月份生成待处理清单，字段至少包括：

- 日期 / 话题 / 消息线索；
- 内容类型；
- 是否已有入库文件；
- public_level 初判；
- value_stage 初判；
- 下一步：追源 / 脱敏 / 写卡 / 暂缓。

### 第二轮：先处理低风险公开资料

优先补录：

- 公开链接；
- GitHub / 官网 / 文档可追源资料；
- 不含内部信息的方法论；
- 已经明确可公开的实验总结。

### 第三轮：处理群聊讨论和内部实践

只抽象成：

- 通用判断框架；
- 问题拆解方法；
- 风险清单；
- 脱敏后的实验卡。

不公开具体人、具体群聊、真实系统细节。

### 第四轮：补索引和复盘

每轮补录后更新或生成：

- 月度补录清单；
- 待追源列表；
- `needs-review` 清单；
- 可沉淀为 Playbook 的候选项。

## 执行边界

历史补录属于批量整理任务，可能涉及大量群聊内容和公开边界判断；如果一次消息量过大，应分批执行并每批提交一个 commit，避免大包提交不可审计。

推荐批次：

- 按月份：每月一批；或
- 按内容类型：公开资料 / 实验记录 / 风险边界 / Playbook 候选。

## 当前结论

近 2-3 个月历史内容补入库是可行的，但前提是能读取或导出对应话题群历史消息。执行时应先做历史消息清单和去重，再分批补录；公开仓库只保留结构化、脱敏、可审计内容。
