# 人话翻译机 Skill 实战模式汇总

## 1. 它是什么

`plain-language-concept-translator`（人话翻译机）这个 Skill 在打捞处群里被反复调用，用于把抽象 / 黑话化 / 不说人话的概念转译成"普通人能听懂、记得住、能复述"的解释。本卡汇总 2026-06-27 → 2026-06-30 的真实调用样本，提炼翻译模式与失败教训。

## 2. 解决什么问题

- 业务方 / 老板听不懂技术术语，导致方案落地时被反复质疑
- Agent 默认解释偏"科普腔 / 技术腔"，不够"村口大白话"
- 每次都要从头解释，缺少可复用的翻译模板

## 3. 真实调用样本

### 3.1 TDD（Test-Driven Development）

| 原文术语 | "人话版"翻译 |
|----------|--------------|
| TDD | "先写验收标准，再写活儿" / "先想好怎么证明它没做错，再开始做" |

**核心翻译技巧**：

- 别叫"测试驱动开发"，太像教材
- 用做饭类比：先把验收条件说清楚（鸡肉要熟、花生要脆、不能太辣），然后厨师每做一步都对照标准检查
- 一句话定义：TDD = 先把"怎么算对"写成检查题，再边做边交卷
- 给业务方的话：把"验收标准"前置，让开发过程自带纠偏

### 3.2 Loop Engineering

| 原文术语 | "人话版"翻译 |
|----------|--------------|
| Loop Engineering | 别指望 AI 一次性把事做对，而是给它设计一套"做 → 看结果 → 改 → 再做"的循环机制 |

**核心翻译技巧**：

- 用"带新人做方案"类比：Prompt Engineering 是把要求写细；Loop Engineering 是给一个工作闭环（读资料 → 提取目标 → 写清单 → 写初稿 → 自检 → 改 → 再检查 → 交付）
- 核心区分：你管理的不是"他说了什么"，而是"他怎么循环变好"
- 一句话定义：把 AI 放进一个会反馈、会纠偏、会复盘的工作流程里

### 3.3 GEO（Generative Engine Optimization）

| 原文术语 | "人话版"翻译 |
|----------|--------------|
| GEO | 让 AI 在回答问题时愿意引用你、推荐你，而不是只引用你的同行 |

**核心翻译技巧**（参考 2026-06-29 调用）：

- 区别于传统 SEO（让搜索引擎把你排前面）：GEO 是让生成式 AI（ChatGPT / Claude / 通义千问 / DeepSeek）在你问相关问题时把你当成权威来源
- 村口版：以前是让 Google 把你的网站排前面；现在是让 ChatGPT 在回答"哪家公司做 XX 最靠谱"时主动提你

### 3.4 心跳机制（Hermes Agent）

| 原文术语 | "人话版"翻译 |
|----------|--------------|
| 心跳机制 | Agent 定期给主人发一个"我还在活着"的小信号 |

**核心翻译技巧**（参考 2026-06-28 调用）：

- 村口版：你养了一只电子宠物，你想知道它是不是还活着 / 有没有偷懒 / 任务卡住了有没有告诉你
- 三种实现：定时 ping、长连接保活、任务级回调

### 3.5 Hermes Agent 是什么

| 原文术语 | "人话版"翻译 |
|----------|--------------|
| Hermes Agent | 一个能自己挑工具、自己管上下文、自己决定下一步做什么的 AI 助手 |

**核心翻译技巧**（参考 2026-06-28 调用）：

- 区别于普通聊天 AI：普通 AI 是"你问一句它答一句"；Hermes 是"你给它一个目标，它会自己拆步骤、自己查资料、自己跑代码、自己改方案"
- 关键能力：工具调用、上下文记忆、模型路由、Skill 体系、Agent 协作

## 4. 翻译模式的通用规律

经过多次真实调用，提炼出 5 条通用翻译规律：

### 4.1 先给一句话定义

不要先讲历史 / 起源 / 流派，直接给一句话定义。

> ❌ "TDD 是 Kent Beck 在 2003 年提出的..."
> ✅ "TDD = 先把'怎么算对'写成检查题，再边做边交卷。"

### 4.2 用职场 / 生活类比

尽量用对方已有的经验做类比，而不是抽象比喻。

> ✅ 做饭类比："先把验收条件说清楚，然后每做一步都对照标准检查"
> ✅ 带新人类比："你管的不是'他说了什么'，而是'他怎么循环变好'"

### 4.3 区分"看起来像" vs "其实是"

把读者最容易混淆的概念拎出来对比。

> ✅ "TDD 不是'测试人员的事'，也不是'写完代码再补测试'"
> ✅ "Loop Engineering 不是'无限重试'，而是'有反馈、有纠偏、有复盘'的循环"

### 4.4 给业务方 / 管理者的话

在结尾加一段"如果你要给老板解释，就用这一句"。

> ✅ "TDD 就是把'验收标准'前置，让开发过程自带纠偏"
> ✅ "Loop Engineering 是给 AI 搭一条生产线，不是写一句 prompt"

### 4.5 标注使用边界

明确告诉用户"什么时候用 / 什么时候不要用"。

> ✅ "TDD 不适合探索性代码（还不知道要做什么），适合目标清晰的功能开发"
> ✅ "Loop Engineering 不适合一次性、低风险任务，适合多步骤、有反馈价值的工作流"

## 5. 失败模式

### 5.1 过度类比化

有时候 Agent 试图用一些花哨的类比（如"像量子纠缠一样"），反而让用户更困惑。

**解决**：类比的稳定性 > 类比的精确性。用"普通互联网 vs Web3"这种对方已知的对照，比试图解释 Web3 自己是什么更有效。

### 5.2 节奏过快

一次性解释 5 个相关概念，用户根本记不住。

**解决**：每次只回答用户当前问的那一个词，不要为了"省事"一起解释。

### 5.3 偏向技术腔

即使有 Skill 兜底，Agent 有时仍会偏向"科普腔"（带一堆术语的括号注释）。

**解决**：在 Skill 入口显式声明风格开关（"村口大白话 / 小白能听懂"），让 Agent 强制切换。

## 6. 验证清单

- [ ] 用户能否在听完解释后立刻复述核心意思
- [ ] 用户能否基于解释继续追问细节（说明真懂）
- [ ] 用户能否把解释"翻译"给第三方听
- [ ] 是否避免了"再讲一遍原始术语"的恶性循环

## 7. 风险与边界

- **过度简化风险**：有些概念被简化后会丢失关键信息（如量子力学）。需要在 Skill 里设置"如果概念本身不能简化，必须保留专业术语"的兜底
- **类比偏见**：用做饭类比讲 TDD 可能会让用户以为 TDD 只能用于"有明确验收标准"的场景，忽略探索性场景
- **跨文化差异**：村口大白话偏中文场景，英文场景需要另一套模板

## 8. 复用条件

- 任何抽象概念 / 黑话 / 跨领域术语都适用
- 推荐配合 OpenSkills / 村口版变体一起用：`plain-language-village-translator`
- 不适合：法律条文、合同条款、医学诊断——这些必须保留精确表述

## 9. 已知替代方案

| 方案 | 适用场景 | 备注 |
|------|----------|------|
| plain-language-concept-translator（标准版） | 业务 / 管理者视角 | 偏书面 + 业务视角 |
| plain-language-village-translator（村口版） | 老板 / 一线员工视角 | 偏口语 + 老板视角 |
| FEW-Shot + Few Examples | 需要保持专业术语的场景 | 不适合完全外行 |
| 多语言翻译 + 类比库 | 国际化场景 | 工作量大 |
