# Canvas Hand：把 Codex 图像生成改造成“画布标注式可控改图”

## 1. 这是什么

Canvas Hand 是一个面向 Codex 的本地无限画布插件。它用 tldraw 做画布，把 Codex 的图像生成、画布标注、局部修改指令和再次生成串成一个循环：先生成图片，再直接在图上用箭头和文字标出要改哪里，Codex 读取标注后重新绘图，并把新图放回画布。

小红书内容的传播点是“指哪改哪”：把原来需要长文本描述的改图需求，转成可视化标注，降低反复沟通和提示词描述成本。

## 3. 核心观点 / 核心能力

- Codex 不只是“写代码”，也可以通过插件、MCP 和本地服务承载多模态工作流。
- 图像编辑的痛点不是“能不能生成”，而是“能不能让人低成本表达修改意图”。
- 画布标注把人的空间意图显性化：箭头负责定位，文字负责说明变化，AI 负责解释和重绘。
- 本质上这是一个“人类可视化意图层 + Agent 执行层”的交互范式，不只适用于改图。

## 4. 我学到了什么

相比纯 prompt 改图，这个方案的价值在于把需求表达从自然语言扩展到空间标注。很多业务协作中的需求，本来就是“这里改一下、那里换一下、这个流程挪到前面”，如果只靠文字描述，很容易丢上下文。画布式交互可以让 AI 更像一个看得懂批注的执行助手。

## 5. 它是否可信，哪些需要验证

已追到 GitHub README，本体清楚，仓库说明包含安装、使用、目录和本地服务机制。仍需验证：

- 当前 Codex 插件机制在本地环境是否稳定可用。
- 端口 8000 固定占用是否会和现有服务冲突。
- MCP 工具读取标注、选择最新图片、重绘回填的准确率。
- 中文标注、复杂多箭头、多轮迭代时是否稳定。
- 生成图片的版权、商用边界和企业数据输入边界。

## 6. 对个人能力有什么价值

这是一个值得学习的 AI 产品交互样例：不是堆模型能力，而是把“人怎么表达意图”重新设计了一层。对做 AI 落地的人，启发是：很多场景不一定要继续优化提示词，而是要把输入界面、上下文结构和执行动作设计好。

## 7. 对企业 AI 落地有什么价值

- 市场/品牌：可用于广告图、海报、产品图的初稿迭代，特别适合“业务方圈点修改”的沟通方式。
- 产品/设计：可借鉴成产品原型、页面截图、流程图的可视化批注修改。
- ITBP：可作为“可视化需求表达 + Agent 执行”的样板，不局限在图片，也可迁移到流程图、表单、CRM 页面、看板等场景。
- AI 工作流建设：说明本地插件、MCP、画布和模型可以组合成低门槛交互工具。

## 8. 可做的小实验

1. 在隔离本地环境安装 Canvas Hand，验证 Codex 插件加载、画布启动、图片生成和标注改图闭环。
2. 用 3 类样例测试：海报改图、产品图局部修改、流程图/界面截图批注。
3. 记录中文标注、多箭头、多轮迭代、失败恢复的表现。
4. 如果稳定，可进一步抽象出“画布标注式 Agent 工作流”的内部 playbook。

## 9. 风险和边界

- 不应把内部客户图、未公开产品图、业务系统截图直接放进外部模型或公开仓库。
- 生成图像涉及版权、肖像、品牌规范和商用授权，需要按使用场景单独判断。
- 该项目依赖 Codex 插件机制和本地 MCP/服务端口，可能受版本变化影响。
- 目前更适合作为个人效率和交互范式验证，不宜直接承诺为生产级设计工具。

## 10. 当前结论

这条资料的价值不只是“Codex 可以改图”，更重要的是它展示了一种更自然的人机协作方式：人用画布表达空间意图，Agent 负责理解和执行。建议进入小实验阶段，先验证插件可用性和中文标注稳定性，再判断是否沉淀为可复用的视觉批注 Agent 工作流。
