# 飞书 CLI 多用户隔离 + 隐私安全

## 1. 这是什么

打捞处群聊 2026-05-05 12:50~19:35 出现的一次**严肃的安全讨论**：
用户（陈总）发现 Mac mini 上绑定自己飞书凭证的飞书 CLI 工具，
理论上**任何能通过模型对话进入通路的智能体（Hermes / OpenClaw / 其他）
都可以读取自己的飞书数据**——
包括联系人、消息、文档、云盘、知识库、邮件等所有 scope。

用户提出的核心问题：
> "我至少得知道谁在读我的数据，读了什么数据。"
> "把一些权限先收回来，每次用的时候，我来授权。"

bot 给出的最终方案不是"关闭飞书网关"（影响业务），
而是一套**多用户隔离 + OAuth Device Code + scope 拆分**的方案。

用户用一个非常生动的比喻总结了这个设计的本质：
> "嫂子，我是我哥，你来开门"—— 你（智能体）要开门，
> 得确认我（用户）是不是真的有钥匙，而不是因为共享了一把钥匙。

## 3. 核心观点 / 核心能力

### 3.1 风险模型：当前飞书 CLI 共享 token 模式的隐患

```
┌──────────────────────────────────────────────────────────┐
│                  当前（不隔离）模式                          │
├──────────────────────────────────────────────────────────┤
│  陈总（Owner）登录飞书 CLI，token 存在 ~/.hermes/.env        │
│     ↓                                                     │
│  Mac mini 上的所有 Agent（Hermes / OpenClaw / 其他）         │
│  都能用同一份 token 调任何 scope                             │
│     ↓                                                     │
│  任何能 "调通" Agent 的人 → 都能读到陈总的飞书数据            │
└──────────────────────────────────────────────────────────┘

风险：
- 隐私：消息、联系人、云盘、知识库、邮件全部可读
- 凭证泄露：单点失守 = 全部数据失守
- 审计缺失：不知道谁读了什么
- 合规：违反最小权限原则
```

### 3.2 目标态：多用户隔离 + OAuth Device Code

```
┌──────────────────────────────────────────────────────────┐
│                  目标（隔离）模式                            │
├──────────────────────────────────────────────────────────┤
│  Agent 要调用飞书 CLI 时：                                  │
│     ↓                                                     │
│  1. 检测当前用户是否已授权                                   │
│     ├─ 已授权：用该用户的 token（独立存储）                  │
│     └─ 未授权：发起 Device Code 授权流程                    │
│     ↓                                                     │
│  2. Device Code 流程：                                     │
│     ├─ 后端生成 device_code + verification_url              │
│     ├─ 用户在浏览器扫码 / 打开链接 → 飞书 OAuth 确认        │
│     ├─ 用户选择 "仅授予必要的 scope"                        │
│     └─ 授权完成后 token 存到该用户的隔离存储                │
│     ↓                                                     │
│  3. 调用记录：                                              │
│     ├─ 每个用户的调用日志独立保留                           │
│     ├─ 记录调用时间、scope、API、操作人                     │
│     └─ 可审计、可回溯                                      │
└──────────────────────────────────────────────────────────┘
```

### 3.3 关键设计决策（用户已表达）

| # | 决策 | 理由 |
|---|------|------|
| 1 | **每个用户的授权独立存储** | "我个人用没差别 + 别人想用要自己授权" |
| 2 | **不放在一起（不分用户隔离）** | "放在一起不会有问题吧？" —— 用户自己否定了，bot 仍建议隔离 |
| 3 | **升级 OpenClaw / Hermes 不影响调用** | 框架升级不应影响已建立的用户授权 |
| 4 | **保留「随时收回」能力** | 用户可主动撤销某个用户的授权，token 立即失效 |
| 5 | **不关闭飞书网关** | 网关是基础设施，关了影响所有用户；只隔离 CLI 凭证即可 |

### 3.4 适用 vs 不适用

**适用**：

- 多用户共享 Mac mini / 服务器场景
- Agent / 智能体框架要调用第三方 SaaS（飞书、企微、钉钉、Notion、Slack 等）
- 团队长 / ITBP 想在团队内推广 AI 工具，又担心凭证安全
- 涉及个人隐私数据（消息、邮件、联系人）的任何自动化

**不适用 / 暂不需要**：

- 纯单人本地使用，且只跑自己的 Agent（但建议也做隔离，应对未来）
- 自动化场景完全不需要用户数据（如纯公开 API 调用）
- 一次性临时脚本（用完即销毁的临时 token 即可）

## 4. 我学到了什么

1. **共享 token 是 Agent 时代最大的安全暗债**：
   任何 Agent 框架默认都会"继承"用户的 token，
   但用户并没有真正授权 Agent 用这些 scope。
2. **"嫂子开门" 是检验身份设计的金句**：
   好的身份验证要让"开门动作"显式、一次、可追溯，
   而不是"已经认识你就直接开门"。
3. **Device Code 流程是 Agent 时代的最佳实践**：
   - 不需要回调地址（适合 CLI / 本地工具）
   - 用户主动授权（满足最小权限）
   - Token 与用户绑定（满足可审计）
4. **审计日志是不可省略的设计**：
   没有"谁读了什么"的日志，所有的"安全设计"都是表演。
5. **凭证存储要按用户隔离而不是按机器隔离**：
   机器隔离是物理的，用户隔离是逻辑的——后者更精细。

## 5. 它是否可信，哪些需要验证

**可信度高**：OAuth Device Code 是 RFC 8628 国际标准，
飞书 / Google / GitHub / Microsoft 都已实现，安全性经过多年验证。

**需要验证**：

- **lark-cli 的 Device Code 流程是否完整支持 scope 拆分**：
  当前 scope 是用户登录时一次性授权的，未来调用新 scope 是否需要重新授权？
- **token 存储的加密方式**：本地 SQLite / JSON 加密？
  macOS Keychain 集成？明文存储风险？
- **审计日志的保留策略**：日志保留多久？谁有权访问？
  GDPR / 个保法合规性？
- **Token 撤销机制**：用户能否主动撤销？撤销是否立即生效？
- **跨设备同步**：用户在 Mac mini 上授权后，
  在 MacBook 上是否需要重新授权？
- **多用户并发场景**：多个用户同时调飞书 CLI 时是否串扰？

## 6. 对个人能力有什么价值

- **安全意识升级**：从"我自己用没风险"升级到"任何共享场景都要做隔离"。
- **架构设计能力**：理解 OAuth Device Code 流程，
  能套用到任何"本地工具 + 第三方 SaaS"的场景。
- **合规意识**：理解最小权限、可审计、显式授权的原则。
- **跨场景迁移**：同样的设计可以套用到：
  - Notion / Slack / GitHub CLI
  - 内部系统的 SSO
  - 任何"本地 Agent 调云服务"的场景

## 7. 对企业 AI 落地有什么价值

- **企业内部推广 AI 工具的前置条件**：
  没有这套隔离设计，企业不敢让 AI 工具碰飞书 / 企微 / 邮箱等敏感数据。
- **多 Agent 协作的安全基础**：
  未来企业内部会有多个 Agent 跑在同一台机器上，
  没有隔离就没有"Agent A 不会读到 Agent B 用户的飞书"的保证。
- **审计与合规**：金融、医疗、政企客户对"AI 访问了哪些数据"有强制审计要求。
- **降低 ITBP 的焦虑**：
  ITBP 在推广 AI 工具时最大的担忧是"出安全事故怎么办"，
  这套设计直接回应了这个担忧。

## 9. 风险和边界

- **凭证迁移成本**：存量系统改造为多用户隔离需要一次性投入。
- **用户体验**：每次新增 scope 都要重新授权，可能让普通用户困惑。
  需要做"一次性全授权 + 按需增删"的引导。
- **存储安全**：本地 SQLite 文件如果被其他进程读取，token 仍然泄露。
  需要加密 + Keychain 集成 + 文件权限三位一体。
- **终端调用的「身份不明」问题**：用户在终端直接调 lark-cli 时，
  Agent 不知道是谁——这是设计上的固有限制，
  解决方法是要求所有调用都通过 Agent，再由 Agent 加注 user 标识。
- **过度隔离的成本**：完全隔离会让"团队共享文档"这类协作变得困难，
  需要做"团队授权"+"个人授权"的双层设计。
- **不要因为设计完美而推迟上线**：MVP 可以先用"全用户共享一个 token + 审计日志"，
  后续再升级到"每用户独立 token"，避免设计瘫痪。

## 10. 当前结论

飞书 CLI 多用户隔离是一个**必须在 Agent 时代解决的基础设施问题**。

OAuth Device Code + 用户独立 token + 审计日志 = 最小可用方案。

MVP 阶段可以先用：
- 每用户独立 JSON 存储 token
- 全量审计日志
- 简单 scope 划分

成熟阶段再加：
- Keychain 集成 + 加密存储
- 细粒度 scope 动态授权
- 跨设备同步 + 撤销机制

**不要等设计完美再上线**——共享 token 的风险每天都在放大。
