# Colleague Skill Runtime：把"蒸馏同事"从能力做成"上岗能力"

## 1. 这是什么

这是打捞处群聊 2026-06-01 → 2026-06-02 凌晨连续讨论中，从一个普通黑客松作品选题碰撞出来的产品方向：**Colleague Skill Runtime（同事 Skill 运行时）**。

它解决的不是"如何把一个同事蒸馏成 AI"，而是行业普遍卡壳的下一段：

> **蒸馏完成后，怎么把这个"同事能力"无缝切入 IM 群聊的工作流、保留上下文、设置边界、并在必要时交还本人？**

## 3. 核心观点 / 核心能力

### 3.1 行业现状：都在讲"蒸馏"，但都不讲"上岗"

调研当时发现的方向关键词都很热：Employee Digital Twin、Context Graph of Work、知识图谱蒸馏。但绝大多数项目卡在"蒸馏完成后怎么用"：

- 蒸馏出来的"数字员工"在网页里 → 用户不会为问一句话专门打开网页
- 蒸馏出来的"知识图谱"在文档里 → 真实问题仍然发生在 IM 群聊里
- 蒸馏出来的"专家助手"挂在 Slackbot → 但用户还得把上下文搬过去

结果就是：很多"数字员工"项目，演示时很炫，一上线就没人用。

### 3.2 Colleague Skill Runtime 的核心定义

> **当员工在 IM 中 @ 某个同事时，系统自动调用该同事授权装载的 Skill，基于当前对话上下文先进行问题补全、风险判断、初步回复或转人工。Skill 的能力范围、可回答边界、升级条件都由本人预先配置，并由本人持续反馈优化。**

村口版：

> 你不是凭空造一个"假同事"。是这个同事自己把一部分常用能力装进去，告诉 AI 哪些能帮我答、哪些不能答。别人 @ 我的时候，AI 先帮我看一眼，简单的先处理，复杂的再叫我本人。

### 3.3 三个差异点

1. **本人授权装载**：不是公司强行"蒸馏员工"，而是员工自己选择装哪些能力。例如主管可以装载：新人销售报价前置检查、常见产品配置问题、客户信息补全要求、报价风险提醒、常见流程说明、审批催办类回复、CRM 填写规范。但不装载：客户机密信息、跨部门纠纷、个人判断题。

2. **工作流里的上下文继承**：用户不需要把问题从群聊复制出去再粘给 AI。AI 本身就嵌在 IM 上下文里，读取的就是当下那段对话——这是普通知识库 / 网页 AI 助手给不了的体验。

3. **活人兜底 + 自我进化**：AI 接不住、风险高、信息不足时，交还本人。本人的反馈回流到 Skill 配置里，持续校准边界。这比"AI 独立闭环"更适合企业内部落地。

### 3.4 一个真实场景的运行示例

> 新人销售在群里 @ 主管："客户要这个报价，我这样报行不行？"
> 主管的数字分身先判断信息完整性 → 发现缺客户分级信息 → 提示新人补充 → 新人补完后再次 @ → 数字分身基于历史报价规则给出风险提醒 + 参考区间 → 同时 @ 主管本人拍板。

整个过程没有跳出 IM，没有让新人搬运上下文，主管只在"最后拍板"和"异常升级"时被叫到。

## 4. 我学到了什么

- **"无缝切入 + 不丢上下文"是 IM AI 助手的胜负手**：单纯做"网页版 AI 助手"已经输给 Slack / 飞书原生机器人无数次了，因为真实问题天然发生在群里。
- **"活人兜底"是企业场景里被反复证明的落地前提**：纯闭环 AI 在企业里要么权限太小（答不上），要么权限太大（乱答背锅）。让员工自己配置边界 + 设置升级条件，是真正可用的设计。
- **"蒸馏"不是终点，运行时才是产品**：行业里讲 Distill 讲了两年，真正缺的不是蒸馏算法，是蒸馏完之后 Skill 怎么进入工作流。这给了"同事 Skill 运行时"一个清晰的产品定位。

## 5. 它是否可信，哪些需要验证

- **可信度高**：定位和痛点是真实从用户工作场景里撞出来的，不是 PPT 推演；OpenSkills + Colleague.skill 已经在做前半段（蒸馏），运行时空缺是真实可见的产品机会。
- **需验证**：
  - **Skill 配置成本**：让员工自己装载并维护 Skill 边界，UI 是否足够轻量？首批落地人群是技术管理者还是销售主管？
  - **上下文继承的工程实现**：IM 群聊里"当前会话上下文"在不同平台（飞书 / 企微 / Slack / 钉钉）的语义粒度差异很大，哪家最容易先跑通？
  - **活人兜底的触发条件**：什么情况下升级到真人，规则要清楚，否则容易被滥用（什么都升级 = 数字分身白做）或漏升级（答错了没人接）。
  - **合规与隐私**：在企业 IM 里读取群聊上下文生成 Skill 的合规边界、撤回机制、误用审计，都需要和法务/合规一起设计。

## 6. 对个人能力有什么价值

- 重新理解"Agent / Skill"的最小落地单元不再是"对话窗口"，而是"嵌入式工作流节点"。
- 学习用"本人授权 + 边界配置 + 升级条件"三件套来设计企业内部 AI 助手，比"通用助手"更可能被采纳。

## 7. 对企业 AI 落地有什么价值

- 给"数字员工"赛道一个比"AI 助手"更精确的产品切法：从"蒸馏 → 上岗"的最后一公里。
- 第一个落地场景建议选有强"师傅带新人"文化的部门（销售实施、客户成功、HR 招聘）——场景里"主管被新人 @"本身就是高频真实问题。
- 落地前必须先沉淀一份"Skill 边界清单 + 升级条件模板"，否则容易做成"什么都答"的助手。

## 8. 可做的小实验

1. **两周实验**：拿销售一组或实施一组，挑 1 位主管做志愿者，把该主管最近 50 条 IM 回复整理成 Work Skill（高频问题的标准回复模式）+ Persona（措辞风格、风险判断风格），让新人以 @ 形式触发。
2. **记录指标**：新人平均完成一次基础问题（不需要主管亲自答）的时间从 30 分钟降到多少？主管被 @ 的次数从每天多少次降到多少次？
3. **失败前置条件**：如果 Skill 升级到本人的触发条件没设计好，会出现"什么都不敢答"或"什么都敢答"两种极端，需要先列黑名单/白名单机制。

## 9. 风险和边界

- **数据合规**：把真实群聊上下文喂给 Skill 生成与运行，需要事先确认是否纳入个人信息出境 / 内部数据使用授权；建议在飞书租户内做私有部署而不是公网 API。
- **误判背锅**：数字分身答错的场景，必须有明确的"非本人责任"和"升级到本人"链路，否则员工不愿意授权装载。
- **幻觉继承**：如果 Skill 的回复模板来自历史消息，而历史消息本身有错，那么数字分身会把错继承并放大。需要版本控制和回归测试。
- **组织协同阻力**：部分老员工可能担心"被 AI 替代"而抗拒授权；需要把 Skill 包装成"帮我挡重复问题"而不是"替我做判断"。

## 10. 当前结论

> **Colleague Skill Runtime 是一个真实可落地的产品方向**，把行业里讲烂了的"蒸馏同事"补上"上岗运行时"这一段。它的差异点不是技术，而是产品定位：**本人授权 + 上下文继承 + 活人兜底**。
> 
> 下一阶段值得做的：用一个真实的小部门（建议销售一组或实施一组），用 2 周时间跑一个最小可用的内部 demo，验证 Skill 边界设计、升级机制、活人反馈闭环是否真的能减少主管被打扰的次数。
