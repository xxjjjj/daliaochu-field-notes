# Agent 自查流程：当用户说"我记得装了 X 工具"时怎么找

## 1. 这是什么

打捞处群聊 2026-06-30 用户说"有个 edm 的工具我记得安装了"，Agent 在没有明确线索的情况下，**用三轮检索确认现状**，给出"有两条线 + 还有一种可能"的判断。

本卡把这套"用户模糊 → Agent 多线索自查 → 给出可选清单"的工作流沉淀下来，给其他 Agent 案例参考。

- 飞书群 message_id：`om_x100b6b059daa4cbcc1b9e22a776b74a`（2026-06-30 09:34，Agent 响应）

## 3. Agent 自查流程（三轮）

### 3.1 第一轮：本机可执行命令扫描

**目标**：验证用户说的工具是不是本机命令行工具。

**做法**：

```bash
which edm focussend resend sendmail mailx msmtp 2>/dev/null
ls /usr/local/bin/ /opt/homebrew/bin/ | grep -iE "(edm|mail|smtp|send)"
```

**结论**：当前机器里没找到 `edm` / `focussend` 这类可执行命令。

**判断**：用户说的"EDM 工具"大概率不是一个命令行工具（至少不是以 `edm` 为可执行名的）。

### 3.2 第二轮：系统级邮件工具盘点

**目标**：本机是否有其他邮件 / SMTP 类工具。

**做法**：

```bash
which msmtp mailx sendmail postfix
brew list | grep -iE "(mail|smtp|edm)"
```

**结论**：

- 找到 `msmtp`：偏底层的 SMTP 发信工具
- 找到 `mailx`：传统命令行发邮件工具

**判断**：本机邮件发送能力是有的，但都不叫"EDM"，所以用户说的不是本机命令。

### 3.3 第三轮：业务历史线索反查

**目标**：用户记忆里的"EDM 工具"是不是业务系统 / 供应商工具。

**做法**：

```bash
# 在飞书历史 / 笔记 / 索引里搜
grep -r "EDM" ~/.hermes/notes/ 2>/dev/null | head -20
grep -r "Focussend\|询盘云 EDM" ~/.hermes/ 2>/dev/null | head -20
```

**结论**：

- 之前讨论过询盘云 CRM / EDM / Focussend 相关情况
- 询盘云自带 EDM / 邮件能力
- 第三方 Focussend EDM 服务也在用
- 需要区分三种费用：询盘云 CRM 费用 / 询盘云 EDM / Focussend EDM

**判断**：用户可能记的是业务系统 / 供应商工具，不是本机命令。

## 4. 给用户的回复结构

最终回复给用户的是"三种可能 + 主动问要不要继续追"：

```
我查了一下，你说的"EDM 工具"大概率不是一个叫 edm 的命令行工具。
现在能确认的有两条线：
1. 系统里有 msmtp（底层 SMTP 发信工具，给脚本 / Agent 用）
2. 系统自带 mailx（传统命令行发邮件工具）
另外我翻到之前业务线索：
你之前查过询盘云 CRM / EDM / Focussend 相关情况。
里面提到过：
- 询盘云自带 EDM/邮件能力；
- 第三方 Focussend EDM 服务；
所以如果你说"我记得安装了一个 EDM 工具"，可能指：
- 本机 msmtp
- 业务系统 Focussend / 询盘云 EDM

如果你想，我可以继续往两个方向查：
- 本机配置里有没有 SMTP / EDM 账号配置
- 飞书历史里查 Focussend / 询盘云 EDM 的入口和负责人
```

**关键技巧**：

1. **先否定一种可能**（命令行工具），降低用户混淆
2. **再列已确认的事实**（msmtp / mailx），让用户知道 Agent 不是空手答复
3. **再列历史关联线索**（询盘云 / Focussend），帮用户回想
4. **最后主动问要不要继续**，避免 Agent 无限挖

## 5. 这套流程的核心原则

### 5.1 先列事实，再判断，再追问

不要一上来就猜"你是不是指 X"。先客观列出"我查到了什么"，让用户确认 / 否定 / 补充。

### 5.2 多线索交叉，不依赖单一来源

- 本机命令（系统层）
- 系统工具盘点（环境层）
- 历史讨论反查（记忆层）

三层线索分别尝试，互相验证。

### 5.3 主动暴露"我不确定"

Agent 不假装"我找到了"，而是明确说"我查到了 X，没找到 Y，可能 Z"。

### 5.4 把决策权交回用户

最后总是问"要不要继续往某个方向查"，让用户决定下一步，而不是 Agent 自己做决定。

## 6. 对其他 Agent 的复用价值

### 6.1 通用场景

- "我之前让谁建了一个 X" → 本机 + 飞书群 + 笔记三层反查
- "我电脑里有个 Y" → which + find + locate
- "我注册过一个 Z 服务" → 飞书历史 + 笔记 + 浏览器历史

### 6.2 跨场景模板

```
用户模糊表达 → Agent 三轮检索：
1. 本机 / 当前环境扫描（which / ls / find / brew list）
2. 系统 / 平台工具盘点（list installed / list configs）
3. 历史 / 记忆反查（grep 笔记 / 搜飞书 / 翻邮件）

回复结构：
- 已确认的事实（"我找到了 X、Y"）
- 没找到的明确否定（"没找到 Z"）
- 历史关联线索（"之前聊过 A、B"）
- 主动追问（"要不要继续往 C 方向查"）
```

## 7. 对 BP / ITBP 的实用价值

- **跨人协作**：当用户记不清"我之前把 X 给了谁 / 谁帮我建了 Y"时，Agent 可以用这套流程帮找回
- **资源盘点**：离职 / 审计场景下，用这套流程快速定位"这个人 / 这个项目涉及了哪些系统"
- **培训 AI 助手**：把这套流程写到 Prompt 里，让 AI 助手学会"先查再说，不要乱猜"
- **减少幻觉**：明确"我没找到" 比 "好像在 X 里见过" 重要得多

## 8. 可做的小实验

1. **场景演练**：让 Agent 接收"我记得装了一个邮件群发工具"这类模糊指令，演练三层检索
2. **检查清单化**：把这套流程写成 Skill，让其他 Agent 直接调用
3. **失败案例收集**：收集"Agent 一上来就猜错"的反例，反向完善这套流程
4. **关键词库建设**：把常见的"EDM / CRM / RPA / Bot"等业务词维护成反查关键词表

## 9. 风险和边界

- **隐私边界**：第三轮"历史反查"会翻用户的历史记录，要先确认有权限
- **过度检索**：用户只是随口一问时，不要把三层都跑完，先快速确认就行
- **判定标准模糊**：什么算"找到"需要明确，否则容易把"grep 到了但只是字符串相似"当成"找到了"
- **时效性**：业务工具会迭代，昨天提到的 Focussend 今天可能换了名字，要定期更新关键词库

## 10. 公开边界

- 涉及的具体工具名 / 业务线名已脱敏（用"询盘云 / Focussend"代替实际业务方）。
- 不暴露具体的检索命令、私域路径、内部账号信息。
- 这套自查方法本身是公开方法论，可直接复用。
