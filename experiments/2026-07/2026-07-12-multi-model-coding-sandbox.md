# 多模型编码助手与安全沙盒小实验

## 1. 测试背景

《人民公园说AI》文字稿提出两个值得验证的判断：

1. 国产模型接入 Claude Code / Codex / OpenCode 等成熟编码生态后，可能在大部分日常任务中具备较高性价比。
2. AI 编码助手的风险边界应从“是否授权某个模型”转为“任务是否在可控沙盒内执行”。

## 2. 测试假设

- 对非敏感、边界清晰的代码/文档任务，GLM/Kimi 等国产模型接入成熟 coding agent 生态后，可能满足 70% 以上的日常辅助需求。
- 使用独立目录或虚拟机隔离任务材料，可以显著降低误读全盘、误改文件、泄露上下文的风险。
- 企业内部更适合沉淀“任务级路由规则”，而不是让普通用户手工判断每次该用哪个模型。

## 3. 测试对象

- 工具链候选：Claude Code、Codex、OpenCode、ZCode、飞书 CLI。
- 模型候选：GLM、Kimi、MiniMax，以及当前可用闭源模型作为对照。
- 测试任务：非敏感脚本整理、文档结构化、配置文件审阅、固定 workflow 修改。

## 4. 测试步骤

1. 准备一个不含客户数据、不含内部密钥的 sandbox 目录。
2. 选择 3 个任务：文档整理、简单代码修改、自动化配置检查。
3. 分别使用闭源模型与国产模型接入 coding agent 工具执行。
4. 记录以下指标：
   - 完成质量；
   - 权限打断次数；
   - 是否误读/误改边界外文件；
   - token/费用；
   - 人工复核时间；
   - 是否能形成可复用 workflow。
5. 输出任务路由建议：哪些任务适合低成本模型，哪些任务必须强模型或人审。

## 5. 输入材料

- 非敏感代码样例；
- 非敏感文档样例；
- 模拟配置文件；
- AI 工具运行日志。

## 7. 失败点 / 异常点

待观察：
- 模型是否因为上下文不足乱改；
- 工具是否频繁要求权限确认；
- 国产模型是否能稳定调用工具；
- 是否存在路径越界、命令风险、数据泄露风险；
- 结果是否需要大量人工返工。

## 8. 是否值得继续

值得。这个实验直接服务于企业内部 AI 工具选型、CRM AI 化模型层设计、Hermes/飞书自动化安全边界设计。

## 9. 下一步动作

- 先查清各工具对外部模型的接入方式与限制；
- 选一个真实但低风险的内部任务做首轮验证；
- 将结果整理为模型路由与沙盒使用 playbook。
