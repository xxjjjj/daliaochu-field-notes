# 跟单提效｜收款预警降噪与自动催款 MVP

## 1. 测试背景

打捞处群聊 2026-05-11，用户希望从历史聊天记录 + 会议纪要中抽取"跟单提效｜自动催款"这条业务需求，并填入飞书 Bitable 项目表。这是一个典型的"Agent 从分散信息源抽取结构化业务需求"实验。

## 2. 测试假设

1. Agent 能从多源信息（聊天 + 会议 + 文档）抽取结构化业务需求。
2. 抽取结果能直接写入飞书 Bitable 项目表，且字段完整、便于后续跟进。
3. 抽取过程能被复用为 Skill，下次有新需求时直接套用。

## 3. 测试对象

- **业务场景**：外销 + 医疗内销的收款流程
- **核心痛点**：
  - 财务逾期后提醒滞后
  - 销售手工台账容易遗漏
  - 未开票明细需手动查 NC
  - 现有预警把付款在途 / 状态未同步 / 内部未认领 / 放单决策等误判为客户未付款风险
- **输入源**：飞书群聊历史、相关会议纪要、《外销收款协议基准日期讨论材料》
- **输出**：飞书 Bitable 项目表 `JMHmbA8qdaByl3s67jzcUhuBnXc` 一条新增记录

## 4. 测试步骤

1. 用户给出 Bitable 项目表链接 + 需求关键词"跟单提效 / 自动催款"
2. Agent 调用飞书消息搜索 + 文档搜索，拉取所有相关聊天和会议记录
3. Agent 调用 MCP Bitable 工具读取项目表 schema，理解字段
4. Agent 把信息按"需求定位 + 核心痛点 + 业务场景拆分 + 协议分类 + 建议 MVP + 负责人 + 对接人"等维度结构化
5. Agent 调用 MCP Bitable 工具新增一条记录
6. Agent 回写群聊确认 + 返回 record_id

## 5. 输入材料

- 项目表：`global-intco.feishu.cn/base/JMHmbA8qdaByl3s67jzcUhuBnXc?table=tblITzPItaY9YlDu&view=vewhAzoAcq`
- 飞书群聊搜索结果（多个）
- 会议纪要：未单独提供，靠 Agent 从聊天里推断
- 参考文档：《外销收款协议基准日期讨论材料》

### 6.2 需求定位的关键判断

> 不是简单"催款机器人"，而是"**收款风险降噪与放单决策辅助**"。

> 当前已有收款预警，下一步应先做**降噪、提前预警**，再推进自动化催款。

这是非常重要的产品定位：用户原本只说要"自动催款"，Agent 通过抽取发现真正的痛点是"预警太多、太杂、太滞后"，所以催款只是末端动作，前置的降噪和提前预警才是核心。

### 6.3 5 个业务场景拆分

```
1. 纯催款
2. 水单核验
3. 放单决策
4. 内部认领
5. 信用风险升级
```

每个场景对应的数据源、规则、阈值都不一样，应该拆成 5 个独立 MVP 而不是 1 个大系统。

### 6.4 6 类外销收款协议

```
- B/L date（提单日）
- before shipment（发货前）
- Against B/L copy（凭提单副本）
- Deposit（预付款）
- OA（赊账）
- L/C（信用证）
```

每类协议的订单量级、风险特征、催款节奏都不一样。

### 6.5 建议 MVP

> 先采集 30-50 个真实事件样本，做"收款预警噪声证据表"，按客户 × 协议类型 × 触发场景 × 实际风险等级 4 个维度标注，作为后续规则引擎的训练集。

## 7. 失败点 / 异常点

### 7.1 会议纪要缺失

用户提到"我的聊天啊会议啊"，但 Agent 实际只能搜到聊天，会议纪要没有提供独立访问通道。这是当前 Agent 的一个能力盲区——会议纪要在很多企业里存在专门的存储空间（如飞书妙记、企微会议），需要在 Skill 设计里显式补上。

### 7.2 协议分类需要二次校对

外销收款协议的分类（6 类）是 Agent 从《外销收款协议基准日期讨论材料》文档里抽取的，但具体每类的"订单量级"数据是 Agent 估算的，需要财务/业务人员二次校对。

### 7.3 MCP Bitable 工具偶发 403

写入 Bitable 时偶尔遇到 MCP 工具 403 错误（App scope 缺失），fallback 到 `lark-cli api` 通用调用 + 设备码授权可正常读写。建议封装 Skill 时优先用 `lark-cli api` 而不是 MCP Bitable 工具。

## 8. 是否值得继续

**值得**。这次实验验证了 Agent 在"业务需求抽取 + 项目表录入"场景的可用性，关键经验：

1. **不要只听用户字面需求**：用户说"自动催款"，但 Agent 通过抽取发现真痛点是"预警降噪"
2. **拆分场景比堆功能更靠谱**：5 个独立场景拆出来后，每个 MVP 都更小、更易验证
3. **证据表优于规则引擎**：先用真实样本标"噪声证据表"，再训练规则，比一开始就写规则更稳
4. **多源信息整合是 Agent 的核心价值**：手工做这件事至少要半天，Agent 5 分钟内出初稿

## 9. 下一步动作

1. **封装 Skill**：`requirement-extraction-from-multimodal-sources`，输入关键词 + Bitable 项目表链接 + 部门，输出结构化项目记录
2. **补齐会议纪要通道**：在 Skill 里集成飞书妙记 / 企微会议 API
3. **建立"业务需求模板库"**：按部门（销售 / 实施 / 市场 / 产品）沉淀常见的痛点分类、场景拆分模板
4. **降噪证据表 MVP**：按本实验建议，采集 30-50 个真实事件样本，标注 4 个维度，作为后续规则引擎训练集
