# AI 编码助手与多模型接入的安全边界

## 背景

近期多模型编码助手、Claude Code/Codex/OpenCode 类工具、国产模型接入成熟 coding agent 生态的讨论增多。它们能显著提升个人和团队效率，但也会带来权限、数据、成本和供应商依赖风险。

## 主要风险

1. **文件访问越界**
   - AI 编码助手可能读取任务无关目录、网盘同步目录或本机敏感文件。
   - 风险不只来自模型，也来自工具层的命令执行能力。

2. **命令执行风险**
   - 自动执行脚本、安装依赖、修改配置、调用外部 API 时，可能造成环境污染或数据外发。

3. **上下文泄露**
   - 将企业代码、客户信息、内部流程、日志、配置等直接交给外部模型，可能触发合规与商业秘密风险。

4. **模型路由不可见**
   - 闭源服务可能存在请求过滤、模型降级、动态路由、限额控制。用户看到的回答不一定来自其以为的模型。

5. **成本失控**
   - 多模型聚合、长上下文、自动循环执行可能造成 token 消耗上升，但未必带来等比例业务产出。

6. **供应商锁定**
   - 如果 workflow 深度绑定单一模型或单一平台，后续限额、价格、地区策略变化会直接影响业务连续性。

## 建议边界

1. **按任务隔离，而不是只按模型隔离**
   - 每个任务建立独立工作目录；
   - 只放入完成任务必需的文件；
   - 不让工具默认读取全盘、网盘、浏览器数据或密钥目录。

2. **高风险任务使用虚拟机或容器**
   - 主机只共享一个低敏目录；
   - 需要上下文时人工复制进去；
   - 任务完成后再人工审核带出。

3. **建立任务白名单**
   - 可自动执行：格式化、摘要、非敏感脚本、测试运行、文档结构化；
   - 需人工确认：生产配置、外部发送、数据库写入、客户数据处理、权限变更；
   - 禁止自动执行：密钥读取、批量删除、未审计外发、绕过权限。

4. **记录关键日志**
   - 模型、工具、输入范围、执行命令、输出文件、人工确认点都要可追溯。

5. **模型层可替换**
   - 企业内部 AI 方案应把业务入口、工作流、模型调用、权限审计拆开，避免被单一模型绑定。

## 当前结论

AI 编码助手的安全核心不是“哪个模型可信”，而是“任务边界是否被工程化控制”。越是要引入多模型和自动化执行，越要先建立目录隔离、权限白名单、人审节点和日志审计。
