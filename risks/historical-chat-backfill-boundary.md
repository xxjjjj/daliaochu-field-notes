# 历史群聊补入库公开边界

## 背景

打捞处话题群可能已经积累了一批尚未进入仓库的资料、讨论、MVP 测试和实验结论。对近 2-3 个月历史内容做补录是有价值的，但历史群聊中往往混有个人表达、内部系统、真实业务判断和未脱敏信息，不能按原文公开。

## 核心原则

历史补录不是聊天记录归档，而是二次研究与脱敏沉淀。

公开仓库中只能出现：

- 外部公开资料链接；
- 自写摘要、判断、业务转译；
- 脱敏后的实验结论；
- 通用方法、Checklist、Playbook；
- 风险边界和待验证问题。

不得出现：

- 群聊原文；
- 具体人员、客户、供应商、内部组织关系；
- 真实业务数据、截图、日志、字段、接口；
- 未授权课件、付费内容、长段原文、逐字稿；
- 未经确认的公司内部方案细节。

## public_level 处理

### public

适用于纯公开资料和自写研究判断，可直接进入公开仓库。

### sanitized

适用于来自群聊讨论、内部实践或实验反馈，但已经完成脱敏和抽象化处理的内容。可以公开，但必须去掉具体业务细节。

### private-only

适用于包含真实业务数据、内部系统截图、客户、流程细节、日志、字段等内容。只可作为本地线索，不进入公开仓库。

### needs-review

适用于公开边界不确定的内容。应先生成待审核清单，不推送到公开仓库，或只推送不含敏感信息的风险说明。

## 补录前检查

每条历史内容入库前至少判断：

1. 是否已有对应 note / experiment / playbook / risk，避免重复入库。
2. 是否含有群聊原文或个人信息。
3. 是否含有客户、内部系统、截图、日志、字段、真实流程细节。
4. 是否只是二手转述，技术类资料是否已追到 GitHub / 官网 / 文档 / Demo。
5. 是否需要标记为 `待追源`、`待验证` 或 `needs-review`。
6. 是否能转成通用方法或脱敏实验结论。

## 批量补录建议

历史补录应按批次进行，每批都保留可审计范围：

- 第一批：近 2-3 个月公开链接和开源资料；
- 第二批：MVP / 实验记录，转成脱敏实验卡；
- 第三批：可复用方法，沉淀为 Playbook；
- 第四批：风险、版权、数据安全、公开边界补充。

每批完成后提交独立 commit，commit message 说明补录范围。
